## 📌 რა არის ეს და რატომ გჭირდება

თუ AI-ს იყენებ კოდის წერისას (Claude Code, Cursor, GitHub Copilot, Windsurf, Cline და ა.შ.), ალბათ შეგხვედრია:

- მოდელი ცდილობს გამოიცნოს რა გინდა — და ცდება
- წერს 1000 ხაზს იქ, სადაც 100 საკმარისი იქნებოდა
- ცვლის მუშა კოდს, რომელიც შენი დავალების მიღმაა
- ამატებს გადაჭარბებულ აბსტრაქციებს და "flexibility"-ს რომელიც არავის უთხოვია
- არ სვამს კითხვას მაშინ, როცა რაღაც გაუგებარია — უბრალოდ აგრძელებს

ეს ფაილი (`CLAUDE.md`) ამ პრობლემებს ჭრის. ჩაწერე შენი პროექტის root-ში და Claude Code-ი ავტომატურად წაიკითხავს ამ წესებს ყოველი სესიის დასაწყისში.

## 🎯 ოთხი ძირითადი პრინციპი

### 1. 🤔 დაფიქრდი სანამ კოდს დაწერ

**არ ვარაუდობ. არ ვმალავ დაბნეულობას. ვაცხადებ trade-off-ებს.**

- დაუშვებელია უხმო ინტერპრეტაცია — თუ მოთხოვნა ორაზროვანია, კითხე
- წარმოადგინე რამდენიმე ვარიანტი — როცა ბუნდოვანებაა, არ აირჩიო ჩუმად
- შეედავე როცა საჭიროა — თუ უფრო მარტივი გზა არსებობს, თქვი
- გაჩერდი როცა დაბნეული ხარ — დაასახელე რა არ გესმის და ჰკითხე

💡 **მაგალითი ცუდი ქცევისა:** მომხმარებელი ამბობს "დაამატე validation". AI ჩუმად წყვეტს რომ ეს ნიშნავს email regex + length check + sanitization, ბევრ რამეს წერს, მერე უკან უნდა დააბრუნოს ნახევარი.

✅ **სწორი ქცევა:** "რომელი ველებისთვის გჭირდება validation? უნდა იყოს client-side, server-side, თუ ორივე? რა error message-ები გინდა?"

---

### 2. ✂️ სიმარტივე უპირველეს ყოვლისა

**მინიმალური კოდი, რომელიც პრობლემას ჭრის. არაფერი spekulaciური.**

- ფუნქცია, რომელიც არ მოგითხოვია — არ დაწერო
- ერთჯერადი კოდისთვის — აბსტრაქცია არ შექმნა
- "flexibility" და "configurability" — მხოლოდ მაშინ, თუ მოითხოვეს
- შეცდომების დამუშავება — მხოლოდ რეალურად შესაძლო შეცდომებზე
- თუ 200 ხაზი 50-ად შეიძლებოდა — გადაწერე

🧪 **ტესტი:** იტყოდა თუ არა გამოცდილი დეველოპერი (Senior) რომ ეს კოდი გადაჭარბებულია? თუ კი — გაამარტივე.

---

### 3. 🔬 ქირურგიული ცვლილებები

**მხოლოდ ის შეცვალე, რაც აუცილებელია. დაასუფთავე მხოლოდ შენი ნაგავი.**

არსებული კოდის რედაქტირებისას:

- არ "გააუმჯობესო" მეზობელი კოდი, კომენტარები, ფორმატირება
- არ refactor-ი იმას, რაც გატეხილი არაა
- შეესატყვისე არსებულ სტილს, თუნდაც სხვაგვარად დაწერდი
- თუ შეამჩნიე dead code რომელიც შენი დავალება არაა — დანიშნე, არ წაშალო

შენი ცვლილებებით orphan-ების შექმნისას:
- წაშალე მხოლოდ ის import/variable/function რომელიც შენი ცვლილებების გამო გახდა გამოუყენებელი
- არ წაშალო ძველი dead code თუ არ გთხოვეს

🧪 **ტესტი:** ყოველი შეცვლილი ხაზი პირდაპირ უნდა მიჰყვებოდეს მომხმარებლის მოთხოვნას.

---

### 4. 🎯 მიზანზე ორიენტირებული შესრულება

**განსაზღვრე წარმატების კრიტერიუმები. გაუსვი loop-ს სანამ არ შესრულდება.**

გადააქციე იმპერატიული დავალებები verifiable მიზნებად:

| ნაცვლად... | გადააქციე... |
|---|---|
| "დაამატე validation" | "დაწერე ტესტები არასწორი input-ებისთვის, მერე გააწარმატე" |
| "გაასწორე bug" | "დაწერე ტესტი რომელიც bug-ს reproduce-ს აკეთებს, მერე გააწარმატე" |
| "Refactor X" | "დარწმუნდი რომ ტესტები მუშაობს refactor-მდე და მერე" |

მრავალნაბიჯიანი დავალებებისთვის — გააკეთე გეგმა:

```
1. [ნაბიჯი] → შემოწმება: [რა შემოწმდება]
2. [ნაბიჯი] → შემოწმება: [რა შემოწმდება]
3. [ნაბიჯი] → შემოწმება: [რა შემოწმდება]
```

💡 **Karpathy-ს ციტატა:** *"LLM-ები საოცრად კარგად loop-ენ კონკრეტული მიზნების მისაღწევად... არ უთხრა რა გააკეთოს — მიეცი წარმატების კრიტერიუმი და უყურე."*
